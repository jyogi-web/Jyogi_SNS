## 【カテゴリE：データバックアップ（Raspberry Pi アーカイブ）】

> Supabase DB / Cloudflare R2 の空き容量を監視し、閾値超過時に自動でチャットログ・画像を Raspberry Pi へ転送して容量を解放する。

| #   | 機能名 | 概要 | ユーザー価値 | 優先度   | 備考 |
| --- | ---- | -- | ------ | ----- | -- |
| E-1 | DB / R2 容量監視 | `pg_database_size()` および R2 List API で使用量を定期取得し、閾値（環境変数で設定）と比較する | 容量不足による障害を事前に防げる | 🔴 P0 | 毎週 cron / Supabase Scheduled Function で実行 |
| E-2 | 動的バックアップ起動判定 | 容量が閾値を超えた場合のみバックアップワーカーを起動。閾値未満はスキップ | 不要なバックアップ処理を省いてコスト・帯域を節約できる | 🔴 P0 | 閾値は `BACKUP_DB_THRESHOLD_PCT` / `BACKUP_R2_THRESHOLD_GB` で管理 |
| E-3 | データエクスポート・Pi 転送 | Supabase から対象テーブルを JSON エクスポート → R2 から画像をダウンロード → tar 圧縮 → GPG 暗号化 → sha256 チェックサム生成 → scp/rsync over SSH で Pi へ転送 | データを安全に Pi へ退避できる | 🔴 P0 | `@aws-sdk/client-s3` (GetObjectCommand)・`web-push` 等既存依存を活用 |
| E-4 | アーカイブ後データ削除 | 転送・整合性検証が成功した対象データを Supabase / R2 から削除して容量を解放する | 転送後に自動で容量が空き、手動削除が不要 | 🟡 P1 | 削除前に Pi 側チェックサム検証を必須とする |
| E-5 | 転送成否通知 | バックアップの成功・失敗を Discord Webhook 等へ通知 | 転送エラーを即座に検知できる | 🟢 P2 | 既存の通知基盤（web-push / Discord）と統合 |
| E-6 | 復元テスト・検証スクリプト | Pi 側で暗号化解除 → tar 展開 → Supabase へ試験インポートする復元スクリプト | バックアップが実際に使えることを保証できる | ⚪ P3 | 年 1 回程度の手動実行を想定 |
